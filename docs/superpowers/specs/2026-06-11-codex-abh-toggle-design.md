# Codex ABH Toggle Design

## Goal

Add a repository-scoped ABH toggle for Codex so a user can explicitly turn ABH mode on or off for the current repository without editing files by hand.

## Context

The repository already exposes read-only Codex guidance through `abh agent setup codex --json`, but it does not provide a command that turns Codex ABH behavior on or off for a specific repository. The user wants a local command that can enable Codex ABH mode for one repository and later disable it, and the solution needs to work with Codex desktop usage rather than only Codex CLI profiles.

## User-Approved Direction

Use repository-local Codex configuration instead of a temporary CLI profile.

- Do not rely on `codex --profile ...` as the primary mechanism because official Codex documentation clearly describes profiles for CLI switching, but does not clearly establish the same per-launch profile flow for Codex desktop.
- Write and remove repository-local `.codex/config.toml` so the same repository behavior is available when opened in Codex desktop.
- Keep the feature Codex-specific for now.

## Requirements

### Functional

1. Add a new top-level command family: `abh codex`.
2. Support `abh codex on` to enable ABH mode for the current repository.
3. Support `abh codex off` to disable ABH mode for the current repository.
4. Support `abh codex status` to report whether Codex ABH mode is currently enabled for the repository.
5. `abh codex on` must create repository-local Codex configuration that instructs Codex to:
   - use the ABH workflow for this repository,
   - begin by running `abh onboarding check --json`,
   - then run `abh next --json`,
   - then run `abh doctor --json`,
   - then run `abh roadmap check --json`,
   - follow verification, audit, close, and memory boundaries already documented in `skills/abh-workflow`.
6. `abh codex off` must remove only the configuration written by `abh codex on`.
7. `abh codex status --json` must return machine-readable state including whether ABH mode is enabled, which file is used, and what action would be taken by `on` or `off`.

### Safety

1. This is a confirmed-write feature, not a silent setup export.
2. `abh codex on --write` without `--confirm` must fail with a structured confirmation error, matching the existing `hooks install` boundary.
3. `abh codex off --write` without `--confirm` must fail with the same confirmation discipline.
4. The command must not overwrite unmanaged repository Codex config.
5. If `.codex/config.toml` already exists and is not ABH-managed, `on` must report a blocker rather than merge or replace the file.
6. If `.codex/config.toml` is ABH-managed, `on` may refresh it and `off` may remove it.

### Documentation

1. Update the README and Codex recipe docs to explain how to enable and disable Codex ABH mode.
2. Update codebase-map / roadmap / task-board / plan docs as needed by the repository’s ABH governance.

## Non-Goals

1. Do not implement temporary Codex CLI profile switching in this slice.
2. Do not implement Claude Code or generic MCP toggles in this slice.
3. Do not merge managed ABH config into an existing user-authored `.codex/config.toml`.
4. Do not auto-launch Codex after enablement.
5. Do not install hooks, create plans automatically, or bypass audit/close gates.

## Design

### CLI Surface

Add:

```text
abh codex on [--write] [--confirm] [--json]
abh codex off [--write] [--confirm] [--json]
abh codex status [--json]
```

Behavior model:

- Default mode is preview, following the same pattern as `abh hooks install`.
- `on` preview reports the managed target path, planned writes, and blockers.
- `on --write --confirm` writes the managed config file.
- `off` preview reports the planned removal if managed config exists, or reports that nothing needs to be removed.
- `off --write --confirm` deletes the managed config file.
- `status` is read-only and reports current repository Codex ABH state.

### File Ownership

Managed path:

```text
.codex/config.toml
```

Managed content strategy:

- Write a complete ABH-managed `.codex/config.toml`.
- Mark the file with a clear ABH-managed comment so ownership detection is explicit.
- Store the ABH instructions in `developer_instructions`, because Codex documentation states that `developer_instructions` are inserted before `AGENTS.md` and can carry repo-specific workflow guidance.

Example shape:

```toml
# ABH MANAGED CODEX CONFIG
# Generated by `abh codex on`; rerun that command to refresh this file.

developer_instructions = """
Use `skills/abh-workflow` for this repository.
Before choosing implementation work, run:
- `abh onboarding check --json`
- `abh next --json`
- `abh doctor --json`
- `abh roadmap check --json`
Follow ABH plan, verification, audit, close, and memory rules for all repository work.
"""
```

This slice intentionally owns the full `.codex/config.toml` file rather than generating partial fragments. That keeps ownership simple and avoids unsafe TOML merging rules.

### Managed / Unmanaged Detection

Rules:

- Missing `.codex/config.toml`: `on` may create it.
- Existing `.codex/config.toml` with ABH managed marker: `on` may refresh it; `off` may delete it.
- Existing `.codex/config.toml` without ABH managed marker: `on` is blocked; `off` reports that ABH mode is not enabled and must not remove the file.

### Module Placement

Add a new module for Codex config management rather than mixing it into `agent_setup.py`.

Recommended files:

- `abh/codex_setup.py` or `abh/codex.py` for preview/write/remove/status logic.
- `abh/commands.py` for command-contract definitions.
- `abh/cli.py` for argparse wiring and JSON/human handlers.
- `tests/test_cli.py` for end-to-end CLI coverage.
- `tests/test_command_contracts.py` for command contract coverage.

Reasoning:

- `agent_setup.py` is currently a read-only export surface.
- This feature is a confirmed-write repository configuration surface, closer to `hooks.py` than to `agent_setup.py`.
- Splitting it into its own module preserves the current read-only/write boundary distinction.

## Data Flow

### `abh codex on`

1. Resolve repository root.
2. Inspect `.codex/config.toml`.
3. Build preview payload with writes and blockers.
4. If `--write --confirm` is not present, return preview only.
5. If blocked by unmanaged config, raise structured ABH error.
6. Ensure `.codex/` exists.
7. Write managed config file.
8. Return the same preview payload with write mode metadata.

### `abh codex off`

1. Resolve repository root.
2. Inspect `.codex/config.toml`.
3. If no managed config exists, return preview/status saying disabled.
4. If file is unmanaged, do not delete it.
5. If `--write --confirm` is present and file is managed, remove the file.
6. Do not remove the `.codex/` directory in this slice; only manage the owned config file.

### `abh codex status`

1. Resolve repository root.
2. Inspect `.codex/config.toml`.
3. Return whether ABH mode is enabled, disabled, blocked by unmanaged config, or absent.

## Error Handling

Expected user-visible cases:

- `abh codex on --write` without `--confirm` -> structured confirmation error.
- unmanaged `.codex/config.toml` already exists -> blocker error on write; preview shows blocker.
- `abh codex off --write` without `--confirm` -> structured confirmation error.
- `off` when nothing is managed -> no-op preview/result, not an exception.

## Testing

### CLI Regression Tests

Add tests covering:

1. `codex status --json` reports disabled when no `.codex/config.toml` exists.
2. `codex on --json` previews a managed write without creating files.
3. `codex on --write --json` fails without `--confirm`.
4. `codex on --write --confirm --json` writes managed `.codex/config.toml`.
5. A second `codex on --write --confirm --json` refreshes managed config safely.
6. `codex on --write --confirm --json` refuses to overwrite unmanaged `.codex/config.toml`.
7. `codex off --json` previews removal when managed config exists.
8. `codex off --write --json` fails without `--confirm`.
9. `codex off --write --confirm --json` removes managed config.
10. `codex off --write --confirm --json` does not remove unmanaged `.codex/config.toml`.

### Contract Tests

Add command contract coverage for:

- `codex.status` as read-only with no confirmation requirement.
- `codex.on` as non-read-only with confirmation `--write --confirm`.
- `codex.off` as non-read-only with confirmation `--write --confirm`.

## Open Questions Resolved

1. Codex-only or multi-agent? Codex-only for now.
2. Temporary profile or repository config? Repository config.
3. Launch Codex automatically? No.
4. Output shell command / env vars? No longer relevant after switching to repository config.
5. Workflow strength? Include setup guidance, required initial checks, and verification/audit command expectations.

## Why This Design

This approach matches the user’s actual desktop usage, preserves ABH’s explicit write boundary model, and fits the repository’s existing pattern:

- read-only setup export in `agent_setup.py`,
- confirmed writes in dedicated write surfaces like `hooks.py`,
- structured JSON envelopes and command contracts,
- no hidden merge logic against user-authored configuration files.
